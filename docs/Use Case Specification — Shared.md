UC-SCH-001 — اختيار المدرسة (Public App)
Primary Actor: مستخدم التطبيق (ولي أمر / طالب / معلّم / مدير مدرسة) Scope: التطبيق العام (Public App) Goal: تحديد المدرسة كسياق (School Context) قبل تسجيل الدخول.
Preconditions
1. التطبيق يعمل بوضع “عام” ويتطلب اختيار مدرسة قبل الدخول.
Trigger
* فتح التطبيق، أو فتح مسار تسجيل الدخول في التطبيق العام.
Main Success Scenario
1. يعرض النظام واجهة اختيار المدرسة.
2. يختار المستخدم أسلوب تحديد المدرسة عبر أحد الخيارين:
    * البحث باسم المدرسة ثم اختيارها من النتائج، أو
    * إدخال كود المدرسة مباشرة.
3. عند البحث بالاسم:
    1. يُدخل المستخدم اسم المدرسة (كليًا أو جزئيًا).
    2. يعرض النظام قائمة بالمدارس المطابقة.
    3. يختار المستخدم مدرسة من القائمة. (اعتماد مباشر)
4. عند إدخال الكود:
    1. يُدخل المستخدم كود المدرسة.
    2. يتحقق النظام من صحة الكود.
    3. عند نجاح التحقق، يعرض النظام نافذة منبثقة (Dialog) تتضمن: اسم المدرسة + الشعار + العنوان (إن وُجد).
    4. يضغط المستخدم “تأكيد”.
5. يعتمد النظام المدرسة المختارة ويثبتها كسياق (school_id + اسم المدرسة).
6. ينتقل النظام إلى واجهة تسجيل الدخول (UC-AUTH-001).
Alternate Flows
* A1 — لا توجد نتائج بحث:
    1. يعرض النظام رسالة تفيد بعدم وجود مدارس مطابقة.
    2. يسمح بتعديل نص البحث وإعادة المحاولة.
* A2 — كود مدرسة غير صحيح:
    1. يرفض النظام الكود.
    2. يعرض رسالة تفيد بأن الكود غير صحيح.
    3. يسمح بإعادة إدخال الكود.
* A3 — تعذر الاتصال/التحقق:
    1. يعرض النظام رسالة مناسبة.
    2. يتيح إعادة المحاولة.
* A4 — المدرسة غير مفعّلة أو غير متاحة:
    1. يرفض النظام اعتماد المدرسة كسياق.
    2. يعرض رسالة تفيد بأن المدرسة غير مفعّلة حاليًا.
    3. يسمح للمستخدم باختيار مدرسة أخرى.
* A5 — إلغاء التأكيد بعد نجاح التحقق (مسار الكود):
    1. يعرض النظام نافذة التأكيد (Dialog).
    2. يختار المستخدم “إلغاء”.
    3. يبقى المستخدم في شاشة إدخال الكود دون تثبيت السياق.
Postconditions (Success)
* تم تحديد المدرسة كسياق معتمد للدخول.
* أصبح المستخدم على واجهة تسجيل الدخول.
Postconditions (Failure / Minimal Guarantee)
* لم يتم إنشاء أي جلسة دخول.
* لم يتم تثبيت سياق مدرسة غير صالحة.

UC-AUTH-001 — تسجيل الدخول (Login)
Primary Actor: مستخدم التطبيق (ولي أمر / طالب / معلّم / مدير مدرسة) Scope: التطبيق العام أو الخاص Goal: تسجيل الدخول وفق نوع التطبيق والدور ضمن سياق المدرسة عند الحاجة.
Preconditions
1. إذا كان التطبيق عامًا: تم تحديد المدرسة مسبقًا (UC-SCH-001).
2. يمتلك المستخدم بيانات الدخول وفق نوع الحساب:
    * وليّ الأمر: رقم الهاتف + كلمة المرور.
    * الطالب/المعلّم/المدير: الرقم المدرسي/الكود + كلمة المرور.
Note
* في التطبيق الخاص تكون المدرسة محددة مسبقًا ولا يُنفّذ UC-SCH-001.
* تطبيق الطالب/ولي الأمر يعرض اختيار نوع الحساب داخل شاشة الدخول (Student/Parent).
* تطبيق المعلّم/الإدارة لا يعرض اختيار دور (الدور معروف من نوع التطبيق).
Trigger
* فتح واجهة تسجيل الدخول.
Main Success Scenario
1. يعرض النظام واجهة تسجيل الدخول.
2. يعرض النظام اسم المدرسة وشعارها في شاشة تسجيل الدخول لتأكيد السياق (Public/Private).
3. في التطبيق العام: يتيح النظام للمستخدم خيار “تغيير المدرسة” قبل إتمام تسجيل الدخول (ينفذ عبر A4).
4. حسب نوع التطبيق:
    * تطبيق الطالب/ولي الأمر: يختار المستخدم نوع الحساب (ولي أمر / طالب).
    * تطبيق المعلّم/الإدارة: يتجاوز النظام خطوة اختيار النوع (نوع الحساب ضمنيًا).
5. يُدخل المستخدم بيانات الدخول المناسبة:
    * ولي الأمر: رقم الهاتف + كلمة المرور.
    * طالب/معلّم/مدير: الرقم المدرسي/الكود + كلمة المرور.
6. يضغط المستخدم زر “تسجيل الدخول”.
7. يتحقق النظام من صحة بيانات الدخول ضمن سياق المدرسة.
8. عند النجاح:
    * يتم إنشاء جلسة دخول فعّالة.
    * ينتقل المستخدم إلى الواجهة المناسبة حسب التطبيق/الدور.
Alternate Flows
* A1 — نوع الحساب غير محدد (تطبيق الطالب/ولي الأمر فقط):
    1. يمنع النظام إتمام تسجيل الدخول.
    2. يطلب من المستخدم تحديد نوع الحساب.
* A2 — بيانات دخول غير صحيحة:
    1. يرفض النظام تسجيل الدخول.
    2. يعرض رسالة “بيانات الدخول غير صحيحة”.
    3. يسمح بإعادة المحاولة.
* A3 — الحساب غير مفعّل أو موقوف:
    1. يرفض النظام إتمام تسجيل الدخول.
    2. يعرض النظام رسالة توضّح حالة الحساب (غير مفعّل / موقوف).
    3. يتيح النظام للمستخدم وسيلة للتواصل مع إدارة المدرسة (UC-SCH-011).
    4. يمنع إنشاء جلسة دخول.
* A4 — تغيير المدرسة قبل تسجيل الدخول (Public App):
    1. يختار المستخدم “تغيير المدرسة”.
    2. يعيد النظام المستخدم إلى اختيار المدرسة (UC-SCH-001).
    3. بعد اختيار مدرسة جديدة، يعود المستخدم إلى واجهة تسجيل الدخول مع تحديث اسم المدرسة كسياق جديد.
    * قاعدة: تغيير المدرسة يحذف سياق المدرسة المحلي فقط (ولا يُعتبر “تسجيل خروج”).
* A5 — المدرسة غير محددة (Public App):
    1. يعيد النظام المستخدم إلى اختيار المدرسة (UC-SCH-001).
* A6 — المدرسة غير مفعّلة (تم اكتشافها عند المصادقة):
    1. يرفض النظام إتمام تسجيل الدخول.
    2. يعرض رسالة تفيد بأن المدرسة غير مفعّلة حاليًا.
    3. يمنع إنشاء جلسة دخول.
* A7 — لا يوجد اتصال بالإنترنت أثناء محاولة الدخول:
    1. يعرض النظام مؤشر “غير متصل” (UC-OFF-070).
    2. يمنع تنفيذ عملية تسجيل الدخول (لأنها تتطلب خادم).
    3. يتيح زر “إعادة المحاولة” عند توفر الاتصال.
Postconditions (Success)
* تم إنشاء جلسة دخول.
* تم فتح الواجهة المناسبة وفق التطبيق/الدور.
Postconditions (Failure / Minimal Guarantee)
* لم يتم إنشاء جلسة دخول.
* لم يتم تغيير سياق المدرسة إلا إذا أعاد المستخدم اختيارها صراحةً.

Phase 2A — أساسيات التطبيق المشتركة
UC-AUTH-010 — تسجيل الخروج (Logout)
Primary Actor: مستخدم التطبيق (ولي أمر / طالب / معلّم / مدير مدرسة) Scope: التطبيق العام أو الخاص Goal: إنهاء جلسة المستخدم الحالية بشكل آمن.
Preconditions
1. المستخدم مسجّل الدخول ولديه جلسة نشطة.
Trigger
* اختيار المستخدم تنفيذ تسجيل الخروج.
Main Success Scenario
1. يختار المستخدم “تسجيل الخروج”.
2. يقوم النظام بإنهاء الجلسة الحالية.
3. يمسح النظام بيانات الجلسة المحلية المرتبطة بالمستخدم.
4. يعيد النظام المستخدم إلى واجهة تسجيل الدخول.
5. في التطبيق العام، يعرض النظام واجهة تسجيل الدخول مع الحفاظ على سياق المدرسة الحالي.
Alternate Flows
* A1 — فشل إنهاء الجلسة:
    1. يعرض النظام رسالة خطأ مناسبة.
    2. يتيح إعادة المحاولة.
Postconditions (Success)
* تم إنهاء جلسة المستخدم.
* لم يعد للمستخدم وصول إلى أي بيانات محمية.
Postconditions (Failure / Minimal Guarantee)
* لا يتم إنشاء جلسة جديدة.
* لا يتم تغيير سياق المدرسة.

UC-SCH-010 — عرض معلومات المدرسة
Primary Actor: مستخدم التطبيق (ولي أمر / طالب / معلّم / مدير مدرسة) Scope: التطبيق العام أو الخاص Goal: تمكين المستخدم من الاطلاع على معلومات المدرسة المرتبطة بالسياق الحالي.
Preconditions
1. تم تحديد مدرسة كسياق حالي (School Context).
2. المستخدم داخل التطبيق (قبل أو بعد تسجيل الدخول).
Trigger
* طلب المستخدم عرض معلومات المدرسة.
Main Success Scenario
1. يعرض النظام معلومات المدرسة المرتبطة بالسياق الحالي.
2. تتضمن المعلومات — إن وُجدت:
    * اسم المدرسة (اسم العرض للمدرسة).
    * كود المدرسة (قابل للنسخ في التطبيق العام).
    * الموقع إن وجد (المحافظة / المديرية / العزلة / الحي).
    * شعار المدرسة.
    * بيانات التواصل المتاحة (البريد الإلكتروني، رقم الهاتف).
3. يمكن للمستخدم الاطلاع على المعلومات دون تعديلها.
Alternate Flows
* A1 — معلومات المدرسة غير مكتملة:
    1. يعرض النظام المعلومات المتوفرة فقط.
    2. لا يمنع عرض الشاشة.
Postconditions
* لا يتم تغيير أي بيانات أو سياق.
* لا يتم إنشاء أو إنهاء جلسة.
Note (Shared UX Rule)
* يعرض النظام “ملخص المدرسة” (الاسم + الشعار) في الواجهة الرئيسية لجميع الأدوار، بينما تفاصيل معلومات المدرسة الكاملة تُفتح عبر هذا الـUse Case (UC-SCH-010).
* “تعديل بيانات المدرسة القابلة للتغيير” هو متطلب خاص بالمدير ويُوثق في URS المدير (ليس هنا).

UC-SCH-011 — التواصل مع المدرسة
Primary Actor: مستخدم التطبيق (ولي أمر / طالب / معلّم / مدير مدرسة) Scope: التطبيق العام أو الخاص Goal: تمكين المستخدم من التواصل مع إدارة المدرسة باستخدام بيانات التواصل المتاحة.
Preconditions
1. تم تحديد مدرسة كسياق حالي.
2. تتوفر بيانات تواصل واحدة على الأقل للمدرسة.
Trigger
* طلب المستخدم “التواصل مع المدرسة”.
Main Success Scenario
1. يعرض النظام وسائل التواصل المتاحة للمدرسة بناءً على البيانات المسجّلة.
2. قد تشمل وسائل التواصل:
    * البريد الإلكتروني.
    * رقم الهاتف الموحّد للمدرسة.
3. عند توفر رقم الهاتف، يتيح النظام طرق التواصل المدعومة عبره، مثل:
    * المراسلة عبر تطبيقات المراسلة المدعومة (مثل واتساب).
    * الاتصال الهاتفي.
    * إرسال رسالة نصية (SMS).
4. يختار المستخدم طريقة التواصل المناسبة.
5. يفتح النظام تطبيق التواصل الخارجي المناسب باستخدام البيانات المتاحة.
Alternate Flows
* A1 — لا تتوفر بيانات تواصل:
    1. يعرض النظام رسالة تفيد بعدم توفر وسائل تواصل حاليًا.
    2. يمنع محاولة التواصل.
* A2 — تعذر فتح تطبيق التواصل الخارجي:
    1. يعرض النظام رسالة مناسبة.
    2. يسمح للمستخدم باختيار طريقة تواصل أخرى إن وُجدت.
Postconditions
* لا يتم إنشاء أو تعديل جلسة المستخدم.
* لا يتم تغيير سياق المدرسة.

UC-OFF-070 — مؤشر عدم الاتصال (Offline Banner) وتعطيل عمليات الخادم
Primary Actor: مستخدم التطبيق (ولي أمر / طالب / معلّم / مدير مدرسة) Scope: التطبيق العام أو الخاص Goal: تنبيه المستخدم عند انقطاع الإنترنت وتعطيل العمليات التي تتطلب خادم.
Preconditions
* المستخدم داخل التطبيق (أي شاشة).
Trigger
* فقد الاتصال بالإنترنت أو فشل اتصال الشبكة.
Main Success Scenario
1. يعرض النظام شريط/مؤشر واضح “غير متصل”.
2. يعطّل النظام أي زر/عملية تحتاج خادم (مثل: تحديث/مزامنة/تحميل جديد/إرسال).
3. عند محاولة المستخدم تنفيذ عملية تتطلب اتصال:
    * يعرض النظام رسالة “يتطلب اتصال”.
4. يتيح النظام إجراء “إعادة المحاولة/Retry” لتحديث حالة الاتصال عند توفر الإنترنت.
5. عند عودة الاتصال:
    * يختفي مؤشر “غير متصل” وتعود العمليات للعمل.
Postconditions
* لا يتم حذف سياق المدرسة أو الجلسة بسبب انقطاع الاتصال.
* يبقى المستخدم قادرًا على تصفح البيانات المحلية إن وجدت (حسب سياسات الـ Offline في الـ SRS).

Phase 2B — الحساب والإعدادات
UC-ACC-010 — عرض الملف الشخصي (View Profile)
Primary Actor: مستخدم التطبيق (ولي أمر / طالب / معلّم / مدير مدرسة) Scope: التطبيق العام أو الخاص Goal: تمكين المستخدم من الاطلاع على معلومات حسابه.
Preconditions
* المستخدم مسجل الدخول.
Trigger
* طلب المستخدم عرض الملف الشخصي / إعدادات الحساب.
Main Success Scenario
1. يعرض النظام معلومات المستخدم الأساسية.
2. تتضمن المعلومات (بحسب نوع الحساب):
    * الاسم الرسمي (للعرض فقط).
    * اسم العرض.
    * رقم الهاتف (إن وُجد).
    * البريد الإلكتروني (إن وُجد).
    * المدرسة (للعرض فقط).
    * للطلاب: الصف + الشعبة + الرقم المدرسي (قابل للنسخ إن لزم).
    * للمعلّم/المدير: الرقم المدرسي/الكود (قابل للنسخ إن لزم).
3. يتيح النظام:
    * تعديل بيانات الحساب (UC-ACC-011).
    * تغيير كلمة المرور (UC-ACC-012).
    * تغيير تفضيلات العرض (UC-ACC-020).
Alternate Flows
* A1 — بعض الحقول غير متوفرة:
    1. يعرض النظام الحقول المتاحة فقط.
Postconditions
* لا يتم تعديل أي بيانات.
* لا يتم تغيير الجلسة أو سياق المدرسة.

UC-ACC-011 — تعديل بيانات الحساب (Edit Account Profile)
Primary Actor: مستخدم التطبيق (ولي أمر / طالب / معلّم / مدير مدرسة) Scope: التطبيق العام أو الخاص Goal: تمكين المستخدم من تعديل البيانات المسموح بها.
Preconditions
* المستخدم مسجل الدخول.
* الحساب فعّال.
Trigger
* اختيار المستخدم تعديل بيانات الحساب.
Main Success Scenario
1. يعرض النظام الحقول القابلة للتعديل فقط (وفق سياسة الدور).
2. يستطيع المستخدم تعديل (حسب السياسة):
    * اسم العرض.
    * رقم الهاتف (إن كان مسموحًا لهذا الدور).
    * البريد الإلكتروني (إن وُجد).
3. يمنع النظام تعديل:
    * الاسم الرسمي.
    * الرقم المدرسي/الكود.
    * الصف والشعبة (للطلاب).
    * بيانات المدرسة.
4. يحفظ المستخدم التعديلات.
5. يتحقق النظام من صحة المدخلات.
6. عند النجاح، يحفظ النظام التغييرات.
Alternate Flows
* A1 — بيانات غير صحيحة:
    1. يرفض النظام الحفظ.
    2. يعرض سبب الرفض.
    3. يطلب تصحيح البيانات ثم إعادة المحاولة.
Postconditions
* تم تحديث البيانات المسموح بها فقط.
Note (UX Rule)
* قد يتم تنفيذ تعديل البيانات عبر Bottom Sheet عند النقر على الحقل داخل الملف الشخصي، بدل شاشة تعديل مستقلة، دون تغيير السلوك الوظيفي (تحقق/حفظ/إلغاء).

UC-ACC-012 — تغيير كلمة المرور (Change Password)
Primary Actor: مستخدم التطبيق (ولي أمر / طالب / معلّم / مدير مدرسة) Scope: التطبيق العام أو الخاص Goal: تمكين المستخدم من تغيير كلمة المرور.
Preconditions
* المستخدم مسجل الدخول.
Trigger
* اختيار المستخدم تغيير كلمة المرور.
Main Success Scenario
1. يطلب النظام كلمة المرور الحالية.
2. يُدخل المستخدم كلمة المرور الحالية وكلمة المرور الجديدة.
3. يتحقق النظام من صحة المدخلات.
4. يحفظ النظام كلمة المرور الجديدة.
Alternate Flows
* A1 — كلمة المرور الحالية غير صحيحة:
    1. يرفض النظام التغيير.
    2. يعرض رسالة مناسبة.
    3. يسمح بإعادة المحاولة.
* A2 — كلمة المرور الجديدة لا تحقق سياسة الأمان:
    1. يرفض النظام التغيير.
    2. يعرض سبب الرفض.
    3. يطلب إدخال كلمة مرور جديدة.
Postconditions
* تم تحديث كلمة المرور بنجاح.
* تبقى الجلسة كما هي.

UC-ACC-020 — تغيير تفضيلات العرض (Light / Dark)
Primary Actor: مستخدم التطبيق (ولي أمر / طالب / معلّم / مدير مدرسة) Scope: التطبيق العام أو الخاص Goal: تغيير ثيم العرض على مستوى الجهاز.
Preconditions
* المستخدم داخل التطبيق.
Trigger
* اختيار المستخدم تغيير الثيم.
Main Success Scenario
1. يعرض النظام خيارات الثيم (فاتح / مظلم).
2. يختار المستخدم الثيم المطلوب.
3. يطبق النظام الثيم فورًا.
4. يحفظ النظام التفضيل محليًا على الجهاز فقط.
Alternate Flows
* A1 — تعذر الحفظ المحلي:
    1. يطبق النظام التغيير للجلسة الحالية فقط.
    2. لا يمنع المستخدم من متابعة الاستخدام.
Postconditions
* تم تطبيق الثيم.
* لا تأثير على الجلسة أو سياق المدرسة.

Phase 2C — مؤشرات السياق الأكاديمي العام (Read-only)
UC-CTX-060 — عرض السنة الحالية والفصل الحالي والمؤشر الزمني
Primary Actor: مستخدم التطبيق (ولي أمر / طالب / معلّم / مدير مدرسة) Scope: التطبيق العام أو الخاص Goal: تمكين المستخدم من رؤية سياق السنة الدراسية الحالية والفصل الحالي، مع مؤشر “كم تبقى حتى نهاية الفصل” كمعلومة عرض عامة.
Preconditions
1. المستخدم داخل التطبيق (عادة في الشاشة الرئيسية).
2. سياق المدرسة معروف.
3. تتوفر بيانات السنة/الفصل الحالي (من الخادم عند الاتصال أو من آخر بيانات محفوظة).
Trigger
* فتح الشاشة الرئيسية (Home/Dashboard).
Main Success Scenario
1. يعرض النظام:
    * اسم السنة الحالية (مثال: 2025/2026).
    * اسم الفصل الحالي (الأول/الثاني/الثالث).
2. يعرض النظام “المتبقي حتى نهاية الفصل” بإحدى الصيغ:
    * عدد الأيام المتبقية، و/أو
    * شريط تقدّم (Progress) للفصل.
3. تبقى المعلومات للعرض فقط (Read-only) لجميع الأدوار.
Alternate Flows
* A1 — تعذر جلب المعلومات وعدم وجود بيانات محفوظة:
    1. يعرض النظام حالة “غير متاح حاليًا” بدلًا من قيمة خاطئة.
* A2 — المستخدم غير متصل بالإنترنت:
    1. يعرض النظام مؤشر “غير متصل” (UC-OFF-070).
    2. يعرض آخر قيم محفوظة إن وجدت.
Postconditions
* لا يتم تعديل أي بيانات.
* لا يتم تغيير الجلسة أو سياق المدرسة.
Note (Role-specific Navigation)
* إذا نقر المدير على هذا المؤشر يمكن توجيهه إلى شاشة إدارة السنة/الفصول (ضمن URS المدير) — هذا الانتقال ليس جزءًا من هذا الـUse Case (العرض فقط مشترك).
